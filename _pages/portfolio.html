[설계 선택과 시간 제약]

이 이슈는 배포 이벤트마다 Process ID를 도입해 스키마·코드를 전면 수정하는 방식과, 기존 구조를 유지한 채 계획서를 시작점/다음 계획서를 종료점으로 보는 쿼리 방식 두 가지 중에서, 마감 일정 때문에 후자를 택한 선택의 결과였습니다. 설계–수정–구현을 여러 번 반복하면서 초기 기획과 실제 구현 사이의 간극을 크게 겪었고, 이후로는 요구사항과 배포 단위를 먼저 명확히 정의한 뒤 설계를 시작해야 한다는 점을 체감했습니다.

[채택한 방식의 한계와 다음 단계]

계획서 범위로 레코드를 조회하는 방식은 추가 개발 없이 동일 PR의 재배포·롤백 이력을 한 번에 모아 볼 수 있다는 장점이 있지만, 배포 상세 화면을 열 때마다 해당 범위를 매번 다시 조회해야 하므로 장기적으로 이력 데이터가 쌓이면 DB 부하가 커질 수 있는 구조입니다. 다음 사이클에서는 Process ID 도입처럼 배포 단위를 스키마 차원에서 명시적으로 구분하거나, 조회 컬럼에 인덱스를 추가하여 읽기 부하를 줄이는 방향으로 개선할 계획입니다.

② ELK 기반 배포 위험도 측정 — 도입 검토 후 미채택
빌드 로그 시각화 PoC → 트레이드오프 검토 → 자발적 폐기 결정
시도 관리자 입장에서 배포 위험도를 정량적으로 측정해 상/중/하로 보여주자는 의견에서 시작했습니다. MSA 환경을 가정해 각 프로젝트의 빌드 시 발생하는 로그(WARN/ERROR/FATAL)를 Elasticsearch에 적재하고 Kibana에서 프로젝트별·시간대별로 시각화해, 위험도 측정의 보조 지표로 활용하려 했습니다.
한계 위험도 기준의 보조 지표로 쓰려 했지만 (1) 데이터가 아직 쌓이지 않은 상태라 의미 있는 패턴을 잡을 수 없었고, (2) 학습 프로젝트 규모 대비 ELK 운영 부담이 과했습니다.
결정 프로젝트 방향이 "배포 사전 위험도 예측"에서 "배포 이력 관리"로 전환되며 ELK는 범위를 벗어났습니다. 구축·운영 난이도를 다시 평가해 미채택을 결정했습니다.

[도구 도입 판단의 한계]

돌이켜보면 ELK를 도입하기까지 고려해야 할 전제(데이터 누적량, 운영 부담, 프로젝트 방향성)가 많았는데, 이런 조건들을 충분히 검토하지 않고 "어떤 기술을 쓸지"에 초점을 맞춘 채 바로 구축부터 시작했습니다. 학습 프로젝트 특성상 로그 데이터가 충분히 쌓일 시간도 없고, ELK 스택을 운영·모니터링할 인력과 시간이 제한적인 상황에서, 결국 위험도 지표로 활용할 만큼의 의미 있는 패턴을 얻지 못했고 운영 비용만 커질 수 있는 구조였다는 점을 뒤늦게 깨달았습니다.

앞으로는 새로운 도구를 도입할 때 "이 기술이 의미 있으려면 어떤 데이터 규모·운영 인력·프로젝트 목표가 전제되어야 하는가"를 먼저 정의하고, 그 전제가 현재 프로젝트 단계와 맞는지부터 확인한 뒤 선택하려고 합니다. 이번 경험을 통해 기술 자체보다 프로젝트의 목적과 시점에 맞는지부터 따져보는 습관이 필요하다는 것을 배웠습니다.

시도 초기에는 데이터 수집 단계에 Kafka를 두고 수집한 데이터를 토픽으로 전달·적재하는 구조로 설계했습니다.
발견 공공 API는 일 단위 배치로 데이터가 갱신되는 성격이라 실시간 스트림이 의미를 가질 만한 데이터 흐름이 아니었습니다. Kafka 운영 부담(브로커·토픽·offset 관리)에 비해 얻는 이득이 거의 없는 오버엔지니어링이라고 판단했습니다.
결정 Crontab 기반 일 단위 배치로 수집 구조를 전환했습니다. 불필요한 인프라 리소스 소비를 제거하고 시스템 복잡도를 낮춰, 파이프라인 운영 안정성을 확보했습니다.
회고
[오버엔지니어링에서 단순함으로]

처음에는 교육 과정에서 배운 아키텍처를 그대로 따르면서 Kafka를 쓰는 게 당연하다고 생각했지만, 하루 한 번 갱신되는 배치 데이터라는 현실과 맞지 않는 오버엔지니어링이라는 점을 뒤늦게 인정하게 됐습니다. 이미 구축한 사항을 빼는 결정이 망설여졌지만, 보여주기용 기술 스택이 아니라 현재 데이터 흐름에 맞는 단순한 구조가 더 중요하다고 판단해 Crontab 기반 일 단위 배치로 전환했습니다.

② RabbitMQ 한계 인식 + Redis Stream 개선 방향
선택한 도구의 한계를 정확히 인식하고 다음 진화 경로를 설계
선택 이벤트 기반 트리거 구조에서 메시지 브로커로 RabbitMQ를 선택했습니다. 실시간 Alert 기반 비동기 트리거 구현에 적합하다는 점이 근거였습니다.
한계 RabbitMQ는 메시지 소비 후 소멸하는 구조라, Consumer 장애 발생 시 트리거 메시지가 유실될 가능성이 있습니다. 재학습 트리거 같은 중요 이벤트는 한 번 놓치면 추적이 어렵습니다.
개선 영속성이 있는 Redis Stream으로 전환하면, Consumer가 다운돼도 메시지가 스트림에 남아 복구 후 재처리가 가능합니다. 다음 사이클에서 적용할 개선 방향으로 정리했습니다.
회고
[선택 시점의 시야 한계]

RabbitMQ를 고를 때는 "실시간 비동기 트리거에 적합하다"는 장점만 보고 시작했지만, 운영을 떠올려 보니 장애가 발생하면 메시지가 그대로 소실되어, 한 번 누락된 재학습 트리거는 복구가 불가능해진다는 한계를 뒤늦게 인식했습니다. 중요한 이벤트일수록 장애 시 재처리·추적 가능성이 더 중요하다는 점을 고려하면, 단순히 "비동기 처리에 잘 맞는다"만으로는 충분한 선택 기준이 아니라는 것을 배웠습니다.

[대안 설계와 평가 원칙의 전환]

이 한계를 보완하기 위해 다음 사이클에서는 메시지가 스트림에 영속되고 장애 후에도 재처리가 가능한 Redis Stream을 대안 경로로 설계해 두었습니다. 앞으로는 메시지 브로커를 포함해 어떤 도구를 선택하든 "정상 동작 시 장점"뿐 아니라 "장애·스케일 변화 시 어떤 식으로 망가지는지"까지 같이 보는 것을 아키텍처 설계의 기본 습관으로 삼으려 합니다.
