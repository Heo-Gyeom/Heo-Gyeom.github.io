처음 기각했던 선택지가 사실은 충분히 가능한 안이었다.

처음엔 A안(연결을 풀에 보관하되, 일정 시간 안 쓴 건 미리 제거하기)을 골랐다가 "내가 제거하는 타이밍과 서버가 끊는 타이밍이 겹치면 의미 없겠다"는 이유로 기각했다. 그런데 나중에 알아보니 AWS의 NAT Gateway는 약 5분(350초) 이상 안 쓴 연결만 끊고, Bedrock도 비슷하게 보수적이다. 내가 "20~30초 안 쓰면 제거"로 설정만 했어도 서버가 끊기 한참 전에 안전하게 정리됐을 일이었다.

결국 A안이 부족했던 게 아니라, 내가 "서버가 언제 연결을 끊는지"를 측정하지 않은 채 추측으로만 기각한 게 문제였다. 정보가 부족했던 것을 옵션 자체의 결함으로 잘못 해석한 셈.

채택한 방식에도 보이지 않는 비용이 있다.

매 요청마다 새 연결을 만들면 HTTPS 보안 협상(TLS handshake)에 시간이 들고, 동시 요청이 많아지면 OS가 사용하는 임시 포트(ephemeral port)가 빠르게 소모돼 시스템 한계에 가까워질 수 있다. 지금은 안정적이지만 트래픽이 늘어나면 다른 문제가 생길 수 있는 구조다.

다음 개선 방향은 두 가지로 보고 있다.

  • 3중 안전장치 구성 — "오래 안 쓴 연결 미리 제거 + 너무 오래된 연결 강제 종료 + 그래도 실패하면 재시도"를 함께 적용
  • AWS 공식 SDK로 전환 — BedrockRuntimeAsyncClient는 연결 관리·재시도·인증을 이미 내장하고 있어, 직접 짠 코드보다 견고함
② 멀티 리전 DR 환경에서의 RPO 검증 부하 테스트
RPO < 1초 SLO 충족 · 데이터 유실 0건 · Aurora Global DB (Seoul-Tokyo)
문제 서울-도쿄 Aurora Global DB 이중화는 구성했지만, 실제 부하 상황에서 RPO < 1초 SLO 충족 여부를 검증하지 않은 상태.
설계 직접 DB 쓰기와 API 경유를 두 경로로 분리해 8개 Phase로 검증. 분리하지 않으면 "복제가 불안정한 건지 vs 요청이 DB에 도달조차 못 한 건지"를 구분할 수 없음. 주 지표는 CloudWatch AuroraGlobalDBReplicationLag.
결과 평균 복제 지연 28~73ms (부하 최대 구간에서 오히려 최저 28ms), 서울/도쿄 데이터 차이 0건, 처리 성공률 97.9%. RPO/유실/스케일링 SLO 전 항목 충족. Aurora 스토리지 레이어 복제는 앱 레이어와 독립적으로 동작함을 실측으로 확인.

테스트 종류 중 빠뜨린 게 있었다.

부하 테스트에는 여러 종류가 있다. 예상 트래픽을 견디는지 보는 Load Test, 한계점을 찾는 Stress Test는 했지만, 트래픽이 0에서 갑자기 500으로 튀는 상황(Spike Test)은 처음부터 고려하지 않았다. 그런데 정작 측정 도중에 그 상황이 우연히 발생했다 — 서버가 자동으로 확장되는 동안 일부 요청이 거절되는 현상이었다. 의도적으로 설계했다면 거절률과 회복까지 걸린 시간을 정확한 숫자로 잡을 수 있었을 것이다.

또 한 번의 실수는 Phase 7에서 "언제부터 측정 데이터를 기록할지" 조건을 정하지 않은 것이다. 서버가 준비되기 전부터 측정이 시작돼 데이터가 오염됐다. 검증 결과의 신뢰성은 데이터 양만큼이나 "어떤 조건에서 측정했는가"의 명확함에서 나온다는 걸 이 실수로 배웠다.

RPO는 통과했지만, 그 옆에서 다른 문제가 보였다.

트래픽이 폭증할 때 서버가 자동으로 늘어나는 약 3분 동안 일부 요청이 거절(503 에러)됐다. Kubernetes의 자동 확장 기능(Karpenter, HPA)이 아무리 빨라도 서버를 새로 띄우는 데는 물리적으로 ~3분이 걸린다. 이건 "트래픽이 들어오는 속도가 서버가 늘어나는 속도보다 빠르다"는 구조적 한계라, 개별 설정을 조정해서 해결될 일이 아니다.

해결 방향은 사용자 요청을 서버가 즉시 처리하지 말고, 메시지 큐(AWS SQS)에 일단 쌓아둔 뒤 서버가 준비된 만큼만 가져가서 처리하는 비동기 구조다. 이렇게 하면 서버가 확장되는 동안에도 요청이 큐에 안전하게 보관되고 거절되지 않는다.

다만 동기(요청→즉시 응답)에서 비동기(요청→큐 적재→나중에 처리)로 바꾸면 새로운 고민이 생긴다 — 같은 요청이 두 번 처리되지 않도록 막아야 하고(중복 처리 방지), 운영도 복잡해진다. 그래서 도입 후 같은 시나리오로 재검증하는 게 다음 단계다.
