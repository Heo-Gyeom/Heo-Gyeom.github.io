① AWS Bedrock AI 챗봇 간헐적 호출 실패
Connection reset 발생률 5~10% → 0%
문제 챗봇 호출 시 Connection reset by peer가 20번 중 1~2번 발생. idle 후 첫 호출에 집중되고, 연속 호출 시에는 거의 없음.
원인 Reactor Netty 기본 커넥션 풀이 Bedrock 서버가 이미 종료한 idle 연결을 재사용 시도 → RST 수신.
해결 ConnectionProvider.newConnection()으로 풀 비활성화 + WebClientRequestException에 한해 지수 백오프 2회 재시도.

A안(maxIdleTime)을 "evict 주기가 서버 종료 타이밍과 겹친다"는 이유로 기각했지만, 사후 검토 결과 NAT GW idle timeout이 350초이므로 maxIdleTime을 20~30초로 잡으면 서버보다 훨씬 먼저 evict해 동일하게 해결할 수 있었음. 기각 근거는 사실 서버 측 idle timeout을 사전 측정하지 않은 상태의 추정이었고, 정보 부족을 옵션의 결함으로 잘못 해석한 셈.

채택한 newConnection()은 매 요청마다 TLS 핸드셰이크 비용과 ephemeral port 소모가 누적됨. 다음 개선 방향은 maxIdleTime + maxLifeTime + Retry 3계층 구성, 또는 커넥션 관리·재시도·SigV4가 내장된 AWS SDK for Java v2(BedrockRuntimeAsyncClient) 전환 검토 중.

② 멀티 리전 DR 환경에서의 RPO 검증 부하 테스트
RPO < 1초 SLO 충족 · 데이터 유실 0건 · Aurora Global DB (Seoul-Tokyo)
문제 서울-도쿄 Aurora Global DB 이중화는 구성했지만, 실제 부하 상황에서 RPO < 1초 SLO 충족 여부를 검증하지 않은 상태.
설계 직접 DB 쓰기와 API 경유를 두 경로로 분리해 8개 Phase로 검증. 분리하지 않으면 "복제가 불안정한 건지 vs 요청이 DB에 도달조차 못 한 건지"를 구분할 수 없음. 주 지표는 CloudWatch AuroraGlobalDBReplicationLag.
결과 평균 복제 지연 28~73ms (부하 최대 구간에서 오히려 최저 28ms), 서울/도쿄 데이터 차이 0건, 처리 성공률 97.9%. RPO/유실/스케일링 SLO 전 항목 충족. Aurora 스토리지 레이어 복제는 앱 레이어와 독립적으로 동작함을 실측으로 확인.

테스트 방법론의 공백. Load / Stress는 설계했지만 Spike Test는 처음부터 고려하지 않았음. 정작 스케일 아웃 진행 중 503이 대량 발생하는 현상이 측정 중 우연히 드러났는데, 의도적으로 설계했다면 분당 거절률과 회복 시간을 정확히 수치화할 수 있었음. Phase 7에서는 측정 시작 조건을 사전에 정의하지 않아 데이터가 오염됨. 검증 결과의 신뢰성은 데이터만큼 측정 조건의 명확함에 달려 있다는 걸 이 실수로 배움.

메시지 큐의 필요성. RPO는 통과했지만 스케일 아웃 진행 중 약 3분간 503이 대량 발생. Karpenter와 HPA가 아무리 빠르게 반응해도 노드·Pod 프로비저닝에는 물리적으로 ~3분이 걸리는, "트래픽 인입 속도 > 인프라 확장 속도"라는 구조적 한계. Producer가 SQS에 즉시 적재하고 Consumer가 준비된 속도에 맞춰 처리하는 비동기 구조로 이 구간을 흡수해야 함. 다만 동기→비동기 전환은 idempotency 설계와 운영 복잡도를 수반하므로, 도입 후 동일 시나리오 재검증이 다음 단계.
