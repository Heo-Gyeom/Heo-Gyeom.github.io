트러블슈팅
① AWS Bedrock AI 챗봇 간헐적 호출 실패
Connection reset 발생률 5~10% → 0%
문제 AI 챗봇을 호출하면 20번 중 1~2번꼴로 갑자기 연결이 끊기는 에러(Connection reset)가 발생했습니다. 1~2분 쉬다가 다시 호출할 때 자주 일어나고, 연속으로 호출할 때는 거의 없는 패턴이었습니다.
원인 백엔드는 한 번 맺은 연결을 재사용하려고 잠시 보관해 두는데, 한동안 쓰지 않으면 AWS Bedrock 서버가 먼저 그 연결을 끊습니다. 그 사실을 모르고 이미 끊긴 연결로 요청을 보내면서 에러가 발생했습니다.
해결 연결을 재사용하지 않고 매 요청마다 새로 맺어 사용 후 즉시 종료하도록 변경했습니다. 더해 일시적인 네트워크 오류는 최대 2회 자동 재시도하도록 안전장치를 추가했습니다.

처음 기각했던 선택지가 사실 충분히 가능한 안이었습니다.

A안(연결을 풀에 두되 일정 시간 안 쓴 건 미리 제거)을 "내 제거 타이밍과 서버 종료 타이밍이 겹치면 무의미하다"는 이유로 기각했지만, AWS NAT Gateway는 5분(350초) 이상 안 쓴 연결만 끊고 Bedrock도 비슷하게 보수적이었습니다. 20~30초로만 잡았어도 서버가 끊기 한참 전에 안전하게 정리됐을 일이었습니다. A안이 부족했던 게 아니라, 서버 측 idle timeout을 측정하지 않은 채 추측으로 기각한 게 문제였습니다.

채택한 방식에도 보이지 않는 비용이 있습니다.

매 요청마다 새 연결을 만들면 TLS 핸드셰이크 비용과 OS 임시 포트 소모가 누적되어, 트래픽이 늘어나면 다른 문제가 생길 수 있는 구조입니다. 다음 개선 방향은 (1) 3중 안전장치 구성(미리 제거 + 강제 종료 + 재시도), (2) AWS 공식 SDK(BedrockRuntimeAsyncClient)로 전환을 검토하고 있습니다.

② 멀티 리전 DR 환경에서의 RPO 검증 부하 테스트
RPO < 1초 SLO 충족 · 데이터 유실 0건 · Aurora Global DB (Seoul-Tokyo)
문제 서울-도쿄 Aurora Global DB 이중화는 구성했지만, 실제 부하 상황에서 RPO < 1초 SLO 충족 여부를 검증하지 않은 상태.
설계 직접 DB 쓰기와 API 경유를 두 경로로 분리해 8개 Phase로 검증. 분리하지 않으면 "복제가 불안정한 건지 vs 요청이 DB에 도달조차 못 한 건지"를 구분할 수 없음. 주 지표는 CloudWatch AuroraGlobalDBReplicationLag.
결과 평균 복제 지연 28~73ms (부하 최대 구간에서 오히려 최저 28ms), 서울/도쿄 데이터 차이 0건, 처리 성공률 97.9%. RPO/유실/스케일링 SLO 전 항목 충족. Aurora 스토리지 레이어 복제는 앱 레이어와 독립적으로 동작함을 실측으로 확인.

테스트 종류 중 빠뜨린 게 있었습니다.

Load·Stress Test는 설계했지만 트래픽이 0에서 500 RPS로 튀는 Spike Test는 처음부터 고려하지 않았습니다. 정작 측정 중에 그 상황이 우연히 발생했고(서버 자동 확장 동안 요청 거절), 의도적으로 설계했다면 거절률과 회복 시간을 정확한 숫자로 잡을 수 있었습니다. Phase 7에서는 측정 시작 조건을 정하지 않아 데이터가 오염됐는데, 결과의 신뢰성은 데이터 양만큼이나 "어떤 조건에서 측정했는가"의 명확함에서 나온다는 걸 이 실수로 배웠습니다.

RPO는 통과했지만, 그 옆에서 다른 문제가 보였습니다.

트래픽 폭증 시 서버 자동 확장(~3분) 동안 일부 요청이 503으로 거절됐습니다. Karpenter·HPA가 아무리 빨라도 서버를 새로 띄우는 데는 물리적으로 시간이 걸리므로, 이건 "트래픽 인입 속도 > 서버 확장 속도"라는 구조적 한계로 개별 설정으로 해결될 일이 아닙니다. 해결 방향은 AWS SQS에 요청을 적재해두고 서버가 준비된 만큼 비동기로 처리하는 구조입니다. 다만 동기→비동기 전환은 중복 처리 방지 등 새 고민을 동반하므로, 도입 후 동일 시나리오 재검증이 다음 단계입니다.

배포맨 — CI/CD 통합 관리 시스템

2025.028주4인 팀 · 팀원

Jenkins · 결재 · 일정 충돌 감지를 하나로 묶어, 배포 계획서부터 사후 보고까지 전 과정을 단일 플랫폼에서 관리합니다.

Java 17 Spring Boot Spring Security QueryDSL PostgreSQL Redis React Jenkins REST API Gemini API
담당한 화면
작업 관리 — 다단계 필터링·정렬·검색
작업 상세 — 타임라인·승인·결과 보고 통합
기여한 부분
작업 관리 — 다단계 필터링 시스템
상태·기간·PR 작성자·프로젝트·배포 결과 등 다수의 조건을 자유롭게 조합할 수 있는 작업 조회 화면을 QueryDSL 기반 동적 쿼리로 구현했습니다. RETRY/ROLLBACK은 일반 배포와 다른 흐름이라 독립 프로세스로 분리했습니다.
작업 상세 — 프로세스 그룹화 및 통합 화면
같은 PR에 대한 재배포·롤백을 PLAN 단위로 그룹화해 배포 흐름을 시간순으로 추적할 수 있게 했습니다. 타임라인, 승인 정보, Jenkins 빌드 로그를 한 화면에서 확인할 수 있도록 통합 구성했습니다.
트러블슈팅
① 배포 프로세스 상태 추적 일치화
동일 PR의 재배포·롤백 흐름을 PLAN 단위로 그룹화
문제 기존 프로세스(계획서 → 배포 → 결과보고/취소/롤백)에서 동일 PR로 재배포·롤백 시 여러 기록이 중복 적재되어, 담당자가 현재 배포 상태를 한눈에 파악하기 어렵고 이력 추적에 시간이 오래 걸리며 휴먼 에러가 발생할 가능성이 있었음.
원인 배포 단위가 PLAN 기준이 아니라 개별 이벤트(배포·취소·롤백) 기준으로만 적재되어, "어느 배포가 어느 프로세스에 속하는지"를 구분할 식별 체계가 없었음.
해결 현재 Deployment가 속한 PLAN을 프로세스 시작점, 동일 PR로 생성된 다음 PLAN(취소·롤백 포함)을 종료점으로 지정해 범위 내 모든 레코드를 조회. createdAt + id 복합 정렬로 동일 시각 배포 순서까지 안정적으로 보장.
회고 보기

두 선택지 사이에서 시간 제약을 우선했습니다.

해결 방향은 두 가지였습니다. (1) Process ID 도입 — 모든 이벤트가 동일 ID로 묶이도록 테이블·코드를 전면 수정하는 방식으로, 가장 명확한 해법이지만 수정 범위가 컸습니다. (2) 프로세스 범위 탐색 — 기존 구조를 유지한 채 PLAN을 시작점, 다음 PLAN을 종료점으로 지정해 쿼리만 수정하는 방식. 마감이 가까워 (2)번을 선택했고, 짧은 시간 안에 동일한 결과를 얻을 수 있는 합리적인 판단이었다고 봅니다.

다만 채택한 방식에는 비용이 따릅니다.

배포 상세 화면을 열 때마다 매번 범위 내 레코드를 다시 조회해야 합니다. 지금 데이터 규모에서는 문제없지만, 이력이 누적되면 DB 조회 부하가 늘어날 수 있는 구조입니다. 다음 사이클에서는 (1)번 방식으로 옮겨가거나 조회 결과를 캐싱하는 보완이 필요합니다.

설계 단계의 어려움도 남깁니다.

멘토 피드백을 반영하며 설계-수정-구현을 반복하다 보니 초기 기획과 많은 부분이 달라졌고, 설계 단계에서 큰 어려움을 겪었습니다. 이를 통해 변경 사항을 시스템 설계 단위에서부터 고려하는 사고방식과 초기 요구사항 정의·팀원 간 협의의 중요성을 직접 체감했습니다.

② ELK 기반 배포 위험도 측정 — 도입 검토 후 미채택
빌드 로그 시각화 PoC → 트레이드오프 검토 → 자발적 폐기 결정
시도 관리자 입장에서 배포 위험도를 정량적으로 측정해 상/중/하로 보여주자는 의견에서 시작. ELK 스택을 docker-compose로 띄우고, MSA 환경을 가정해 각 프로젝트의 빌드 로그(ERROR/WARN/FATAL)에 가중치를 부여해 Elasticsearch에 적재. Kibana에서 프로젝트별·시간대별 카운트를 시각화했음.
한계 위험도 기준의 보조 지표로 쓰려 했지만, (1) 데이터가 아직 쌓이지 않은 상태라 의미 있는 패턴을 잡을 수 없었고, (2) 학습 프로젝트 규모 대비 ELK 운영 부담이 과함, (3) 실시간 로그는 Jenkins 로그를 SSE로 관리자 페이지에 직접 스트리밍하는 방식으로 대체 가능했음.
결정 프로젝트 방향이 "배포 사전 위험도 예측"에서 "배포 이력 관리"로 전환되며 ELK는 범위를 벗어남. 구축·운영 난이도를 다시 평가해 미채택을 결정했음.

"도구를 쓸 수 있다는 것"에 의미를 뒀습니다.

ELK를 시도한 동기에 솔직한 부분이 있습니다. "이런 도구 사용할 줄 압니다"를 보여주고 싶다는 마음이 컸고, 정작 그 도구가 우리 프로젝트 시점에 의미 있는 데이터를 만들 수 있는지는 충분히 따지지 않았습니다. 학습 프로젝트는 데이터가 누적될 시간이 없는데, ELK 같은 관측 도구는 데이터가 쌓인 뒤에야 가치가 생깁니다. 시점이 맞지 않으면 도구는 기술 데모로 남습니다.

다음엔 도구보다 상황과 방향성을 먼저 봅니다.

다음에 비슷한 도입 결정이 오면 "쓸 줄 안다"가 아니라 "이 도구가 의미를 가지려면 어떤 전제(데이터 누적량, 사용자 수, 운영 인원)가 필요한가"를 먼저 정의하고, 그 전제가 현재 프로젝트에 갖춰져 있는지 확인한 뒤 도입을 결정하겠습니다. 이번 경험이 "기술 호기심"과 "프로젝트 적합성" 사이의 거리를 인식하게 해 준 사례로 남았습니다.

그래도 시도에서 남긴 것.

ELK 스택을 직접 구성하고 Elasticsearch 색인 구조·Kibana 시각화 감각을 잡았습니다. 프로젝트 규모가 커지고 운영 데이터가 충분히 누적되는 시점이 오면, 배포 위험도 예측 지표로 ELK를 다시 검토할 만한 출발선을 갖춘 셈입니다.

MLOps 2단계 아키텍처 설계 — 지하철 승객 예측 파이프라인

2024.07 ~ 2024.093개월3인 팀

MLOps 2단계 파이프라인을 직접 설계·구축하며, 각 컴포넌트의 역할과 기술 선정 기준을 스스로 판단하는 경험을 목표로 진행한 프로젝트입니다. 데이터 수집부터 모델 갱신까지의 라이프사이클을 이벤트 기반 자동화로 연결했습니다.

Performance Monitoring — Elasticsearch + Grafana
사용자 피드백 데이터를 Elasticsearch에 적재하고 Grafana로 실시간 시각화했습니다. NoSQL을 선택한 이유는 일관성·무결성보다 가용성과 성능이 더 중요한 데이터 성격이었고, 코멘트 분석 시 Full-text 검색을 활용할 수 있다는 점, Grafana와의 직관적인 연동까지 고려한 결정이었습니다.

사용자 만족도가 30% 이하로 떨어질 때 Grafana Webhook이 RabbitMQ로 메시지를 전달하고, Consumer가 이를 수신해 Airflow API로 재학습 DAG를 트리거하는 이벤트 기반 자동 재학습 구조를 설계했습니다.

① Kafka 기반 수집 구조 → Crontab 배치로 전환
데이터 성격을 다시 보고 오버엔지니어링을 자발적으로 제거
시도 초기에는 데이터 수집 단계에 Kafka를 두고 실시간 스트림처럼 처리하는 구조로 설계했음. MLOps의 일반적인 레퍼런스 구조를 따른 선택.
발견 공공 API는 일 단위 배치로 데이터가 갱신되는 성격이라 실시간 스트림이 의미를 가질 만한 데이터 흐름이 아니었음. Kafka 운영 부담(브로커·토픽·offset 관리)에 비해 얻는 이득이 거의 없는 오버엔지니어링이라는 판단.
결정 Crontab 기반 일 단위 배치로 수집 구조를 전환. 불필요한 인프라 리소스 소비를 제거하고 시스템 복잡도를 낮춰, 파이프라인 운영 안정성을 확보.

"표준 아키텍처"가 곧 "우리 아키텍처"는 아닙니다.

MLOps 레퍼런스를 따라 자연스럽게 Kafka를 넣었지만, Kafka의 강점은 대량의 실시간 이벤트를 버퍼링하고 여러 Consumer가 병렬로 소비하는 시나리오에서 발휘됩니다. 우리 데이터는 하루에 한 번 공공 API에서 가져오는 정적 배치 데이터였습니다. 도구의 강점과 데이터의 성격이 일치하지 않았습니다.

"빼는 결정"이 "넣는 결정"보다 어려웠습니다.

Kafka를 빼겠다고 했을 때 잠깐 망설였습니다. "이미 구축했는데 굳이?", "MLOps 프로젝트인데 Kafka 정도는 들어가야 하지 않나?" 같은 생각이 있었지만, 결국 아키텍처는 보여주기 위한 게 아니라 데이터를 안정적으로 흘리기 위한 것이라는 기준으로 돌아왔습니다. Crontab으로 전환하니 시스템은 더 단순해졌고 운영 부담은 사라졌습니다.

다음에 적용할 기준.

도구를 도입할 때 "이 도구의 강점이 우리 데이터의 성격과 일치하는가"를 먼저 묻는 습관이 생겼습니다. 표준 아키텍처는 출발점일 뿐, 정답이 아닙니다.

② RabbitMQ 한계 인식 + Redis Stream 개선 방향
선택한 도구의 한계를 정확히 인식하고 다음 진화 경로를 설계
선택 이벤트 기반 트리거 구조에서 메시지 브로커로 RabbitMQ를 선택. 실시간 Alert 기반 비동기 트리거 구현에 적합하고, AMQP 표준으로 안정성이 검증되어 있다는 점이 근거.
한계 RabbitMQ는 메시지 소비 후 소멸하는 구조라, Consumer 장애 발생 시 트리거 메시지가 유실될 가능성이 존재. 재학습 트리거 같은 중요 이벤트는 한 번 놓치면 추적이 어려움.
개선 영속성이 있는 Redis Stream으로 전환하면, Consumer가 다운돼도 메시지가 스트림에 남아 복구 후 재처리 가능. 다음 사이클에서 적용할 개선 방향으로 정리.

도구를 고른 뒤에 그 도구의 한계까지 보는 습관을 얻었습니다.

RabbitMQ를 골랐을 때는 "비동기 트리거에 적합하다"는 장점만 봤습니다. 하지만 운영 중 한 가지 시나리오가 떠올랐습니다 — Consumer Pod가 죽은 사이에 도착한 메시지는 어떻게 되는가? RabbitMQ는 기본적으로 메시지를 한 번 소비하면 소멸시키기 때문에, 그 사이에 떨어진 재학습 트리거는 추적할 수 없습니다.

대안을 미리 설계해두는 것의 가치.

당장 운영 문제는 아니지만 "트래픽이 늘거나 Consumer 장애가 잦아지면 분명히 문제가 된다"는 점이 보였습니다. 그래서 Redis Stream으로 전환하는 경로를 미리 설계해뒀습니다. 영속성이 있고 Consumer Group으로 멀티 Consumer를 지원하며, 메시지 ID 기반으로 재처리가 가능하기 때문입니다.

아키텍처를 본다는 것은 "왜 그 자리에 그것이 있는가"를 보는 것.

이 프로젝트를 통해 MLOps 아키텍처가 유기적으로 연결되는 구조를 직접 구축하며, 각 기술이 왜 그 자리에 배치되어야 하고 어떤 인프라 구조가 필요한지를 파악할 수 있었습니다. 데이터 생성부터 모델 갱신까지의 라이프사이클을 이해하고, 복잡한 시스템 설계 시 전체 흐름을 파악하는 능력을 키울 수 있었습니다.
